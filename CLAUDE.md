# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Art / rendering rule (binding)

When a reference implementation exists, port its drawing/animation code verbatim first, then
refactor structure (split, rename, de-globalize) — never redraw visuals from a text
description. The canonical art spec is `SIPPY-GRAPHICS-FIX.md` (logical canvas 420×740, exact
coords/colors/anchors); the approved render code is the prototype at `sippy-prototype.html` (repo
root). **All in-game visuals stay procedural** (canvas draw code, no image assets) — this is why
`giant.js` / `sippy.js` / `hats.js` draw everything. Gemini/AI image tools may ONLY be used for
store & marketing collateral (`store-assets/`), never for runtime game art.

## Source of truth

`SIPPY-BUILD-SPEC.md` is the locked, authoritative build spec; read it before any work. The
prototype that was ported lives at `sippy-prototype.html` (repo root) — port from it, don't reinvent.

The game is **implemented and running on-device** (Milestones 1–4 + 6 done; see the memory file for
current state). Remaining owner-blocked work: RevenueCat IAP (M5) and real Play Games Console IDs.
Do not treat native IAP or Play Games leaderboards as live until those IDs exist.

SIPPY is a shippable Android game: a single-file HTML5 canvas / Web Audio prototype wrapped by
**Capacitor 7** with no bundler — `www/` is served as static files (vanilla JS, no React/Vite/TS).
Package `com.genartstudios.sippy`. Release artifact is a signed `.aab` with Play App Signing.
External service IDs live in `www/js/gameids.js`; the local Play Games plugin is `plugins/playgames/`.

## Working agreement (from spec §3 — binding)

- Game logic lives in `www/js/`. It MUST run in a plain desktop browser with zero native plugins
  (`native.js` provides no-op fallbacks). Test in browser FIRST, device second.
- All gameplay tuning goes in `balance.js`. Never scatter magic numbers.
- All native access goes through `native.js`. No plugin imports anywhere else in game code.
- No `localStorage` / `sessionStorage` anywhere. Use `save.js` (Capacitor Preferences plugin only).
- Audio: dynamic sounds (buzz, slurp, heartbeat) are WebAudio-synthesized; one-shots are `.ogg`
  samples with a synth fallback if a file fails to load.
- Never commit: `google-services.json`, `*.keystore`, RevenueCat keys (use `local.properties` / env).
- Before claiming a milestone done, run its acceptance gate (spec §10).

## Commands

- dev in browser:  `npx serve www`
- sync native:     `npx cap sync android`
- debug on device: `cd android && ./gradlew installDebug`
- release bundle:  `cd android && ./gradlew bundleRelease`

There is no test runner; the "tests" are the per-milestone acceptance gates in spec §10
(e.g. M1: full loop playable via `npx serve www`; M3: synth fallback works with audio files
deleted; M6: sign-in failure never blocks play). Verify against those gates, not a CI suite.

## Architecture (the big picture)

Two hard layering rules define this codebase, both enforced by single chokepoint files:

1. **Native isolation — `www/js/native.js`** is the *only* file that touches Capacitor plugins
   (haptics, ads, IAP, analytics, the local Play Games plugin). Everything else calls the
   facade, which no-ops on web. This is what keeps the game playable in a desktop browser.
2. **Persistence isolation — `www/js/save.js`** is the *only* file that persists data, via the
   Capacitor Preferences plugin. Direct web storage APIs are banned.

Game flow is a state machine in `state.js`:
`title → flyin → landed → drinking → windup → escape → splat → gameover`. `main.js` owns boot,
resize, the RAF loop, and screen orchestration. Renderers are procedural canvas drawing (no image
assets): `giant.js` (procedural giants), `sippy.js` (mascot, including hats anchored to a head
transform), `fx.js` (particles/shake/splat), `hud.js`. `hats.js` is a cosmetic registry mapping
ids → draw functions → IAP SKUs; equipped hats persist via `save.js` and must reappear on the
death/ghost/share scenes. All cosmetics are drawn, never imported as images — same for app icons,
which `tools/make_assets.py` renders procedurally.

The native side adds one **local Capacitor plugin** at `plugins/playgames/` (Kotlin, Play Games
Services v2) because no maintained community plugin exists; all other native capabilities use
off-the-shelf Capacitor plugins (spec §4 plugin matrix — verify current major versions at build
time, don't trust the table blindly). The Android shell (`android/`) is generated by
`npx cap add android` then customized: portrait-locked, edge-to-edge immersive, keep-screen-on.

## Feature wiring constraints (spec §5 — easy to get wrong)

- **Rewarded ad ("Second Wind"):** offered once per splat before game over, max 3/night, no
  banners/interstitials (protect suspense). Decline / no-fill → straight to game over.
- **IAP** via RevenueCat, all non-consumable: `remove_ads_999`, `hat_pack_party_199`,
  `hat_pack_fancy_199`. Play Billing must be v7+.
- **Play Games v2** auto-signs-in; never block gameplay on sign-in failure. Leaderboards
  `best_night_ml`, `closest_call_ms`; 7 achievements. IDs come from Play Console into
  `strings.xml` + a `gameids.js` constants file.
- **Firebase Analytics:** emit *exactly* the spec §5 event taxonomy — no more, no less:
  `sip_end`, `splat`, `night_end`, `second_wind`, `iap_purchase`, `hat_equipped`.

## Audio generation

SFX/voice one-shots are generated via the ElevenLabs MCP (manifest in spec §6), then run through
`tools/normalize_audio.sh` (ffmpeg → mono ogg, ~-14 LUFS, trimmed, each ≤100 KB). Audition each
against its synth placeholder and keep whichever is funnier. The TikTok hook is
`vox_onemoresip` + `slap` + `harp`.

## Owner-provided / out of scope for Claude

These cannot be done in-repo (spec §11): Play/AdMob/Firebase/RevenueCat accounts, the upload

spec §8–§9 — mirror them, don't improvise.
